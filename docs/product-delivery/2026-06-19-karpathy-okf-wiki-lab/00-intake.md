# Karpathy OKF Wiki Lab Intake

## 请求原文

构建一套从需求洞察到产品方案、开发测试、上线验收的 Codex 能力，覆盖多智能体、skill、MCP 和 harness；重新审视体系是否健壮、不冗余；优化后执行一个基于 Karpathy LLM Wiki 和 Google OKF 的本地知识库项目，并提供 HTML 可视化和管理页面。

## 当前假设

- 项目先做本地可运行版本，不连接远程数据库。
- Markdown/OKF 文件是知识源，HTML 页面是浏览和管理辅助界面。
- 发布验收以本地 preview、截图、测试、PDOS preflight 为准，不自动 push、建 PR 或部署线上环境。
- Agent team 采用文件化任务板加运行时 subagent 的模式，而不是常驻后台进程。

## 目标用户

- 主要用户：希望用 Codex 管理长期项目能力和个人/团队知识库的产品负责人。
- 次要用户：未来参与知识库维护的研究、产品、工程、QA、发布和自我改进角色代理。

## 业务目标

- 把“需求到上线”的工作方法沉淀为可复用、可验证的 Product Delivery OS。
- 用一个真实小项目检验该 OS 是否能支持多角色协作、质量门禁、UI 验证和交付验收。
- 形成一个本地可继续扩展的 OKF 知识库样板。

## 成功口径

- PDOS 技能本身通过自测和 skill 校验。
- 知识库项目能完成 OKF lint、单元测试、集成构建、E2E 合同测试、桌面和移动截图测试。
- HTML 页面可在本地服务中打开，支持搜索、筛选、图谱、详情、hygiene 统计和草稿 frontmatter 生成。
- 交付包通过 `validate --ready` 和 release preflight。

## 风险面

- OKF 仍处早期草案，需保留扩展兼容性。
- HTML 页面不能直接写回本地文件，管理能力应明确为辅助生成和检查。
- 多智能体容易冗余，必须用角色、任务、依赖和 handoff 控制边界。
- 截图测试如果有动态时间会不稳定，因此页面草稿时间固定来自数据集日期。
